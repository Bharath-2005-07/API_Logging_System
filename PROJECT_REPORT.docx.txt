1. INTRODUCTION
================================================================================

The digital era has brought unprecedented challenges in tracking, verifying, and 
securing API usage logs. Traditional centralized logging systems face issues with 
data manipulation, lack of transparency, and difficulty in proving authenticity 
without a trusted third party.

This project presents a Secure and Immutable API Usage Logging System that 
leverages blockchain technology to create tamper-proof, permanently recorded, and 
cryptographically verifiable API logs.

1.1 PROJECT OVERVIEW
====================

Our system is a decentralized architecture that combines:
- Ethereum Blockchain for immutable storage
- IPFS for decentralized file storage
- Smart Contracts for automated recording
- Digital signatures (RSA-2048) for authenticity verification
- MongoDB for efficient caching and metadata
- React frontend for user-friendly interaction

Key Features:
• Immutable log recording on Ethereum blockchain
• Digital signatures ensuring authenticity
• Decentralized IPFS storage for data content
• Automated verification with 5-second auto-verify
• Comprehensive billing system ($2 base + $5 for POST requests)
• Payment processing with refund/remaining balance logic
• Rate limiting and access control
• Complete audit trail with timestamps
• Blockchain integration for transparency

1.2 PROJECT SIGNIFICANCE
========================

This system addresses critical real-world problems:
1. **Data Integrity**: Logs cannot be modified after creation (immutability)
2. **Transparency**: All stakeholders can verify logs independently
3. **Auditability**: Complete chain of custody from creation to verification
4. **Accountability**: Digital signatures prove who created each log
5. **Compliance**: Maintains audit-ready records for regulatory requirements
6. **Cost Tracking**: Automatic billing for API usage and monitoring

1.3 SCOPE & LIMITATIONS
=======================

SCOPE:
- API request/response logging
- User authentication and profile management
- Blockchain-based log storage
- Digital signature verification
- Billing and payment processing
- Log verification and retrieval
- Admin dashboard and statistics

LIMITATIONS:
- Ethereum Sepolia testnet only (not mainnet production)
- Requires ETH for gas fees
- IPFS pinning limited to 90 days (without paid service)
- Rate limiting per IP address only
- No batch processing for multiple logs

2. PROBLEM STATEMENT
================================================================================

2.1 EXISTING CHALLENGES
=======================

Traditional API logging systems face multiple challenges:

PROBLEM 1: DATA TAMPERING
- Centralized systems can be compromised
- System administrators can modify logs
- No cryptographic proof of authenticity
- Historical log modifications are undetectable

PROBLEM 2: TRUST & VERIFICATION
- Users cannot independently verify log authenticity
- Disputes over API usage/costs become difficult to resolve
- No way to prove logs haven't been altered retroactively
- Third-party audits are expensive and time-consuming

PROBLEM 3: ACCOUNTABILITY
- Difficult to track who performed what action
- No non-repudiation - users can deny their requests
- No proof of timestamp accuracy

PROBLEM 4: REGULATORY COMPLIANCE
- Many industries require immutable audit trails
- Existing systems lack cryptographic proof
- Compliance demonstrators are manual and prone to error

PROBLEM 5: SCALABILITY & COST
- Existing solutions require expensive infrastructure
- Cloud storage costs scale with data volume
- No transparent pricing model

2.2 SPECIFIC PROBLEM THIS PROJECT SOLVES
==========================================

This project addresses all above problems by:
1. Storing logs on Ethereum blockchain (tamper-proof)
2. Using RSA-2048 signatures for authentic origin proof
3. Implementing decentralized IPFS storage
4. Providing cryptographic verification mechanism
5. Creating transparent, usage-based billing system
6. Enabling independent verification by any stakeholder
7. Maintaining permanent, auditable records

6. CONSENSUS ALGORITHM & JUSTIFICATION
================================================================================

6.1 ETHEREUM'S CONSENSUS MECHANISM
===================================

Current Mechanism: Proof of Stake (PoS)
Transition Date: September 15, 2022 (The Merge)
Previous Mechanism: Proof of Work (PoW) until 2022

6.2 HOW PROOF OF STAKE WORKS
=============================

In Ethereum's PoS (Beacon Chain):

1. VALIDATORS
   - Network participants stake ETH (32 ETH minimum)
   - Validators propose and validate blocks
   - Must maintain high uptime and honesty
   - Penalized (slashed) for misbehavior

2. BLOCK PROPOSAL
   - Random validator selected to propose new block
   - Selection weighted by stake size
   - Proposer includes transactions from mempool

3. BLOCK VALIDATION
   - Committee of validators (128 default) assigned to block
   - Validators attest to block validity
   - 2/3 supermajority required for finality
   - Finality achieved in 2 epochs (384 seconds)

4. REWARDS & PENALTIES
   - Honest validators earn staking rewards (~4% annual)
   - Penalties for: offline, equivocation, other violations
   - Slashing: Up to 32 ETH loss for severe violations

6.3 JUSTIFICATION FOR PROJECT USE
==================================

WHY PROOF OF STAKE FOR OUR PROJECT?

✓ ENERGY EFFICIENCY
  - PoS uses 99.95% less energy than PoW
  - No computational arms race
  - Environmentally sustainable
  - Student-friendly deployment

✓ SECURITY THROUGH ECONOMICS
  - Validator must bond capital (stake) to participate
  - Misbehaving validator loses capital (slashing)
  - Attack cost is very high
  - Economic security is understandable

✓ FINALITY ASSURANCE
  - Blocks finalized in ~6 minutes (2 epochs)
  - Logs stored are permanently secure
  - Reverting would require attacking majority of stake
  - Cost prohibitive for attackers

✓ SCALABILITY ROADMAP
  - Ethereum 2.0 sharding increases throughput
  - Future: 100,000+ transactions per second
  - Better supports large-scale logging systems

✓ DECENTRALIZATION
  - Lower barrier to entry than PoW
  - Can run validator on standard hardware
  - ~900,000 validators as of 2024
  - Highly distributed network

COMPARISON: PoW vs PoS

                    Proof of Work    Proof of Stake
Energy Usage        ~120 TWh/year    ~600 MWh/year
Hardware Required   Specialized      Standard
Attack Cost         Very High        Stake Amount
Confirmation Time   ~10-60 min       ~6 minutes
Decentralization    Lower            Higher
Environmental       Poor             Excellent

6.4 ETHEREUM'S SECURITY MODEL
=============================

For Our Project Specifically:

1. LOG IMMUTABILITY
   - Once stored on blockchain, cannot be changed
   - Would require attacking majority validators
   - Validators have cryptographic commitments
   - Cost of attack >> value of logs

2. TIMESTAMP AUTHENTICITY
   - Block timestamp set by protocol
   - Cannot be manipulated post-hoc
   - Secured by network consensus

3. TRANSACTION FINALITY
   - After 2 epochs (384 seconds), transaction is final
   - Only possible reversal: 51% attack (economically infeasible)
   - Our logs are safe after 384 seconds

4. SEPARATION OF CONCERNS
   - Blockchain handles consensus & storage
   - Our backend handles business logic
   - IPFS handles data storage
   - Each layer secures different aspects

struct APILog {
    bytes32 ipfsHash;      // IPFS content hash (256-bit)
    bytes signature;       // RSA-2048 signature bytes
    uint256 timestamp;     // Block timestamp (seconds since epoch)
    address loggedBy;      // Ethereum address of log creator
    string userId;         // User identifier from database
    string endpoint;       // API endpoint path e.g., "/api/users"
    uint256 statusCode;    // HTTP status code (200, 404, 500, etc.)
    uint256 requestSize;   // Request body size in bytes
    uint256 responseSize;  // Response body size in bytes
    bool verified;         // Verification flag (true/false)
}

STRUCT 2: APIUser
───────────────────────────────────────
struct APIUser {
    address userAddress;   // Ethereum wallet address
    string userId;         // Username from authentication
    uint256 totalRequests; // Count of total logs created
    uint256 totalCost;     // Total cost in wei
    bool isActive;         // Account status (enabled/disabled)
    uint256 createdAt;     // Registration timestamp
}

STRUCT 3: BillingRecord
───────────────────────────────────────
struct BillingRecord {
    string userId;         // User identifier
    uint256 requestCount;  // Number of requests in period
    uint256 totalCost;     // Total cost in wei
    uint256 startDate;     // Period start timestamp

    bool isPaid;           // Payment status
}

9.4 API INTEGRATION WITH BLOCKCHAIN
====================================

9.4.1 LOG CREATION FLOW
───────────────────────

API Call: POST /api/logs/create
Request:
{
    "endpoint": "/api/users",
    "method": "GET",
    "statusCode": 200,
    "requestSize": 256,
    "responseSize": 1024,
    "responseTime": 45
}

Backend Processing:

Step 1: Validate JWT Token
├─ Extract token from Authorization header
├─ Verify signature matches JWT_SECRET
├─ Extract userId from token payload
└─ Return 401 if invalid

Step 2: Generate Log Hash (LoggingService)
├─ Create log object with all data
├─ Serialize to JSON string
├─ Generate SHA-256 hash
├─ Return: 64-char hex string
└─ Example: "a1b2c3d4e5f6..."

Step 3: Create Digital Signature (LoggingService)
├─ Load RSA private key from file
├─ Sign hash with private key (PKCS#1)
├─ Return: base64-encoded signature
└─ Approx size: 344 bytes

Step 4: Upload to IPFS (IPFSService)
├─ Serialize log object to JSON
├─ Connect to Infura IPFS
├─ Upload file
├─ Receive IPFS hash (Qm + 44 chars)
├─ Example: "QmXxX7xXxX7xXxX7xXxX7xXxX7xXxX7xXxX7xXxX7"
└─ Fallback: Mock hash if Infura unavailable

Step 5: Store on Blockchain (BlockchainService)
├─ Create ethers.JsonRpcProvider
├─ Create ethers.Wallet with private key
├─ Create Contract instance (address + ABI)
├─ Call: contract.storeLog(...)
├─ Sign transaction with wallet
├─ Submit to Sepolia network
├─ Wait for confirmation (in 1 block)
├─ Receive transaction hash
└─ Example: "0xab12cd34ef56..."

Step 6: Calculate Billing
├─ Check method type
├─ Base cost: $2.00
├─ Method cost: POST = +$5.00, others = $0
├─ Total: e.g., $7.00 for POST
├─ Convert to cents: 700
└─ Store in billing record

Step 7: Save to MongoDB
├─ Create Log document with:
│  ├─ logHash
│  ├─ ipfsHash
│  ├─ blockchainHash
│  ├─ signature
│  ├─ userId
│  ├─ endpoint, method, statusCode
│  ├─ requestSize, responseSize
│  ├─ verified: false (default)
│  └─ timestamp: new Date()
├─ Save to database
└─ Return document ID

Step 8: Update Billing (MongoDB)
├─ Find current month's billing record
├─ If not exists: create new record
├─ Increment requestCount += 1
├─ Add cost: totalCost += 700 (cents)
├─ Save record
└─ Log: "[BILLING] Updated..."

Step 9: Return Response
Response (201 Created):
{
    "success": true,
    "data": {
        "logHash": "a1b2c3d4...",
        "ipfsHash": "QmXxX7xXxX...",
        "blockchainHash": "0xab12cd34...",
        "signature": "base64encoded..."
    },
    "message": "Log created and stored"
}

Step 10: Auto-Verify (After 5 seconds)
├─ Frontend waits 5 seconds
├─ Calls: POST /api/logs/:logHash/verify
├─ Backend calls: contract.verifyLog()
├─ Blockchain updates: verified = true
├─ MongoDB updates: verified = true
└─ Frontend refreshes table

9.4.2 BLOCKCHAIN INTERACTION CODE EXAMPLE
──────────────────────────────────────────

BlockchainService.js constructor:

const ethers = require('ethers');
const APILoggerABI = require('../utils/APILoggerABI.json');

constructor() {
  // 1. Create provider (read-only)
  this.provider = new ethers.JsonRpcProvider(
    process.env.ETHEREUM_RPC_URL
  );
  
  // 2. Create signer (can sign transactions)
  this.signer = new ethers.Wallet(
    process.env.PRIVATE_KEY,
    this.provider
  );
  
  // 3. Create contract instance
  this.contract = new ethers.Contract(
    process.env.CONTRACT_ADDRESS,
    APILoggerABI,
    this.signer  // Use signer for write operations
  );
}

storeLog method:

async storeLog(ipfsHash, signature, userId, endpoint, 
               statusCode, requestSize, responseSize) {
  try {
    // Call contract function (writes to blockchain)
    const tx = await this.contract.storeLog(
      ipfsHash,           // bytes32
      signature,          // bytes
      userId,             // string
      endpoint,           // string
      statusCode,         // uint256
      requestSize,        // uint256
      responseSize        // uint256
    );
    
    // Wait for confirmation (1 block)
    const receipt = await tx.wait();
    
    // Extract transaction hash
    return receipt.hash;  // "0xab12cd34..."
    
  } catch (error) {
    console.error('[Blockchain] Error:', error.message);
    return null;  // Graceful degradation
  }
}

12.1 THREAT MODEL
================

THREAT 1: DATA TAMPERING
threat Level: High
Attack Vector: Modify logs after creation
Mitigation:
  ✓ Blockchain immutability (logs on-chain cannot change)
  ✓ Digital signatures (tampered data fails verification)
  ✓ Hash comparison (stored hash vs recreated hash mismatch)
  ✓ Event logging (blockchain events are immutable)
Result: Logs are cryptographically protected ✓

THREAT 2: ACCOUNT TAKEOVER
Threat Level: High
Attack Vector: Steal JWT tokens or credentials
Mitigations:
  ✓ JWT tokens expire after 7 days
  ✓ Passwords hashed with bcryptjs (10 rounds, ~100ms per hash)
  ✓ HTTPS in production (encrypts tokens in transit)
  ✓ Secure .env storage (private key not exposed)
  ✓ Rate limiting prevents brute force attacks
Result: Accounts are reasonably protected ✓

THREAT 3: INJECTION ATTACKS
Threat Level: Medium
Attack Vector: SQL/NoSQL injection, XSS
Mitigations:
  ✓ mongo-sanitize library (prevents NoSQL injection)
  ✓ Input validation (whitelist allowed characters)
  ✓ React escaping (prevents XSS by default)
  ✓ Parameterized queries (Mongoose ODM)
Result: Injection attacks are prevented ✓

THREAT 4: DENIAL OF SERVICE (DoS)
Threat Level: Medium
Attack Vector: Spam requests to overwhelm service
Mitigations:
  ✓ Rate limiting (100 requests/15 minutes per IP)
  ✓ Gas costs (100 free logs @ 95,000 gas each before costs add up)
  ✓ Signature verification (computationally expensive to forge)
Result: DoS attacks are mitigated ✓

THREAT 5: BLOCKCHAIN EXPLOITATION
Threat Level: Low (for Sepolia testnet)
Attack Vector: 51% attack, double-spending
Mitigations:
  ✓ Ethereum network consensus (900,000+ validators)
  ✓ Proof of Stake economics (attack costs more than gain)
  ✓ Finality (after 2 epochs, reversals extremely unlikely)
  ✓ For testnet: acceptable risk, mainnet requires additional measures
Result: Blockchain attacks are economically infeasible ✓

12.2 CRYPTOGRAPHIC SECURITY
===========================

ALGORITHM 1: RSA-2048 (Digital Signatures)
───────────────────────────────────────────
Key Size: 2048 bits
Security Level: ~112 bits (equivalent security)
Expected Break Time: > 2^111 operations
Current Status: Secure until ~2030
Recommendation: Plan migration to RSA-4096 by 2030+

Implementation:
├─ Public key: Stored with user profile
├─ Private key: Secured in .env file
├─ Signature: PKCS#1 v1.5 padding
├─ Verification: On blockchain + backend
└─ Threat: None currently, keys managed securely

ALGORITHM 2: SHA-256 (Hashing)
──────────────────────────────
Output Size: 256 bits
Collision Probability: ~2^128 (computationally infeasible)
Current Status: Secure, NIST approved
Expected Lifespan: Secure for 20+ years

Implementation:
├─ Input: Log JSON data
├─ Output: 64-character hex string
├─ Collision Cost: > $10 trillion (per latest research)
└─ Status: Industry standard ✓

ALGORITHM 3: HS256 (JWT Signing)
────────────────────────────────
Algorithm: HMAC-SHA256
Secret Length: Should be >= 32 bytes (256 bits)
Current Status: Secure if secret is strong
Implementation:
├─ Secret stored in .env (JWT_SECRET)
├─ Token includes: userId, email, issuedAt, expiresAt
├─ Expiration: 7 days
└─ Threat: Token compromise = account access
     Mitigation: Use HTTPS, secure storage, token refresh

ALGORITHM 4: BCRYPTJS (Password Hashing)
─────────────────────────────────────────
Salt Rounds: 10 (deprecated, should use 12+)
Hash Time: ~100ms per password (intentionally slow)
Current Status: Secure, industry standard
Recommendation: Increase salt rounds to 12-14

Implementation:
├─ During registration: Hash passwords with bcryptjs
├─ During login: Compare input with stored hash
├─ Brute force cost: 100ms × 1 trillion = 3170 years
└─ Status: Passwords well-protected ✓
